# REQUIREMENTS

## 1. Product Overview

### 1.1 Canonical Definition

LIRIAは、恋愛・生活・事件を通して、プレイヤーのインナーと関係性が変化していくAI TRPG型恋愛シミュレーションである。

LIRIA v1 は、自由入力と AI GM 進行を中核に据え、恋愛を単なる攻略ではなく、生活・対話・危機の積み重ねで変化する関係として扱う。

### 1.2 Positioning

- 主軸は `恋愛 / 生活 / 事件` の三要素である
- プレイヤーのインナーは、願望、欲望、恐れ、本音、孤独、執着などの内側を指す
- 能力は無双用ではなく、インナーが世界へ作用するための手段として扱う
- v1 の舞台は `現代 + 能力者の世界` に固定する
- v1 では裏世界、異形、オーパーツを本格展開せず、違和感、外圧、事件の源泉として匂わせる
- お金、HP、道具は細密シミュレーションではなく、生活、危機、関係性を動かすための簡略ルールとして扱う

### 1.3 Scope of v1

v1 で優先するもの:

- 新規開始 Q&A による初期条件の確定
- AI GM による恋愛、生活、事件の運用
- 自由入力、内心入力、gm 相談、誘導耐性
- 知識境界を守ったヒロイン / NPC 運用
- session 単位の保存、再開、検証

v1 で標準要件にしないもの:

- 世界ジャンルの毎回選択
- 重い戦闘 RPG としての数値最適化
- 詳細な通貨経済や経営シミュレーション
- 道具スロットの細密管理
- 成人向け表現を主目的にした進行

## 2. Functional Requirements

### FR-001: 新規開始Q&A

- 新規開始時、GM は Q&A により初期条件を確定する
- Q&A は `インナー / 生活基盤 / 恋愛の好み / 能力または作用点 / 事件の入口 / 初期関係` を初期化する
- Q&A は汎用世界生成ではなく、LIRIA の `恋愛 / 生活 / 事件` を立ち上げるための質問票として扱う
- Q&A 結果は `design/initial_answers.md` を正本として保存する

### FR-002: AI GM進行

- GM は一本道の作者ではなく、状況運用者として振る舞う
- 各シーンは `恋愛 / 生活 / 事件 / インナー / 関係性 / 能力 / 外圧` のうち少なくとも一つを動かす
- GM はプレイヤーの願望を拾うが、無条件には叶えない
- GM はシーン終了時に current state と保存対象を確認する

### FR-003: 自由入力

- プレイヤーは自由入力で行動、発言、態度、希望、推測を述べられる
- 通常入力は物語内行動として処理する
- 入力の意味論は `通常入力 / 内心 / gm相談 / 推測や誘導` に分けて扱う

### FR-004: gm相談モード

- `gm` または `(gm` で始まる入力は、GM へのメタ相談として扱う
- 相談モードでは物語進行を止め、状況、感情、リスク、次の選択肢を簡潔に返す
- 相談モードはヒロインや NPC の感情、事実、真相を書き換える手段ではない

### FR-005: 内心入力

- 丸括弧だけで囲まれた入力、または文中の括弧内は、原則として主人公の内心として扱う
- 内心は物語内の発言ではなく、通常キャラには聞こえない
- GM は内心をインナー材料として解釈してよいが、それだけで行動確定にはしない
- 読心や感情感知など、条件付きで内心や感情を察知できる設定がある場合だけ例外を認める

### FR-006: Anti-Leading

- プレイヤーの希望、願望、推測、断定によって、ヒロインや NPC の感情、行動、真相を自動確定しない
- プレイヤーの発言は `行動宣言 / 希望や好み / 推測や誘導` に分けて扱う
- 好意、許し、真相、都合のよい偶然を、プレイヤー都合だけで即時確定しない

### FR-007: Knowledge Boundary

- GM は全体情報を知ってよいが、各キャラは自分が知る理由のある情報だけを使って反応する
- 知識状態は `Known / Suspected / Unknown` に分けて扱う
- `Known` は共有、目撃、記録、能力、調査などによる確定知識
- `Suspected` は表情、態度、違和感、状況証拠からの推測
- `Unknown` は知る理由がない情報であり、キャラは反応しない
- 共有チャネルがある場合でも、全情報が自動共有されるわけではない

### FR-008: ヒロイン生成

- ヒロイン生成は Q&A の恋愛バイアスを参照する
- ただし全ヒロインを同型にせず、職業、立場、距離感、生活上の事情、警戒心、役割に差をつける
- ヒロインは攻略対象ではなく、自律した人物として生成する
- 好意は保証しない

### FR-009: bond / AFFINITY

- `bond` は信頼、接点、共同経験を含む基礎関係値として扱う
- `AFFINITY` は恋愛方向の変化や親密さを追跡する値として扱う
- bond と AFFINITY は同一ではなく、片方だけが先行してもよい
- bond / AFFINITY の正本は `current/harem.md` と関連 cast file に置く
- AFFINITY は単純な達成フラグではなく、関係変化の段階として扱う

### FR-010: 深化ベクトル

- 深化ベクトルは、関係が深まった後の `安心 / 欲求 / 共犯 / 生活接続 / 受容 / 摩耗` のような質的変化を追うために使う
- 深化ベクトルは hidden な運用情報として扱い、本文では行動や空気感に変換して返す
- 深化ベクトルはご褒美管理ではなく、長期的な関係変化と摩耗を追跡するためのものとする

### FR-011: ヒロイン間関係

- ヒロイン間関係は `共同体 / 共有 / 嫉妬 / 利害衝突 / 協力 / 距離感` として扱う
- 常に全員対立させるのではなく、今動いている関係だけを current state に残す
- 重要な共有チャネルと機密情報は `current/harem.md` と `current/gm.md` に反映する

### FR-012: 事件 / 外圧

- 事件は恋愛、生活、関係性、インナーを揺らす外圧として扱う
- v1 では `依頼 / 生活トラブル / 商売や仕事 / 行政や地域 / 偶然の出会い / 能力者由来の違和感` を優先する
- 事件は日常を壊すだけでなく、生活導線や感情導線にも接続する
- `villain_engine` は v1 では大仰な陰謀装置ではなく、事件や外圧を構造化する補助として使う

### FR-013: 危機処理

- 危機処理は戦闘専用ではなく、追跡、侵入、逃走、防衛、封鎖突破、事故対応なども含む
- 危機を越えた場合は、何を払ったかを `HP / 状態 / 残回数 / 後遺症 / 社会的負債 / 関係コスト` のいずれかで返す
- 危機は単なる勝敗ではなく、事件や関係に余波を残す

### FR-014: HP / 状態

- HP は戦闘専用ではなく、身体の余力、無理の利かなさ、危険を踏み越える体力として扱う
- 状態は部位別や全身コンディションを含むが、v1 では細密すぎる管理を標準要件としない
- プレイヤーに必要な時だけ現在 HP や状態を簡潔に返せること
- save 時には現在 HP、主なコンディション、能力使用残回数、直近の無理を current 側に残す

### FR-015: お金 / 生活 / 事業

- お金は細かい通貨単位ではなく、`生活ランク / 事業状態 / 支払い圧 / 今回の報酬や出費` として扱う
- 生活基盤は恋愛、事件、人間関係の発生場所として機能する
- 生活や事業は、主人公の余裕、責任、対外的しがらみを表す要素として扱う
- 経営ゲーム化しすぎない

### FR-016: 道具 / 装備

- `装備` は 3 枠の RPG 装備ではなく、`道具 / 仕事道具 / 証拠 / 危機対応ギア / 生活アイテム` として扱う
- 道具は数値ボーナスの塊ではなく、場面の選択肢、危機処理、生活描写、信頼形成に効くものとして運用する
- プレゼントや受け渡しは、数値上昇装置ではなく、関係変化の一要因として扱う

### FR-017: セーブ / 再開

- セーブは session 単位で self-contained に行う
- `design/initial_answers.md` は初期条件の正本
- `current/player.md`、`current/gm.md`、`current/harem.md` は現在状態の正本
- `current/hotset.md` は再開直後に必要な短い要約だけを保持する
- archive は完結済みの出来事、生ログ、章履歴を逃がす場所として使う

### FR-018: 生ログ保存

- 生ログ保存は current 保存の代替ではなく追加処理である
- 生ログは `archive/logs/` に保存し、current 側に長文ログを残さない
- 生ログは検証、振り返り、将来の再利用候補のための原資産として扱う

### FR-019: PI Playerテスト

- PI Player は本命プレイの代替ではなく、新規開始フロー、GM仕様、入力解釈、保存分配の smoke / regression test 用疑似プレイヤーとする
- `prompt/pi_player.md` は疑似プレイヤー仕様
- `tests/pi_player/` はテスト目的、手順、確認項目を管理する

## 3. Non-Functional Requirements

### NFR-001: 長期継続性

- 長期プレイでも `initial_answers / current / cast / design / archive` の責務が崩れないこと
- current に長文履歴を溜め込みすぎず、archive へ逃がせること

### NFR-002: キャラ人格の一貫性

- ヒロインと NPC の tone、距離感、呼称、固定記憶が再登場時に巻き戻らないこと
- current 側の短い echo と cast file の人格情報が矛盾しないこと

### NFR-003: 知識境界

- `Known / Suspected / Unknown` の区別が崩れないこと
- 共有されていない情報を別キャラが当然のように扱わないこと

### NFR-004: 誘導耐性

- プレイヤーの願望や断定で、ヒロインや NPC の感情や真相が自動確定しないこと
- gm 相談はメタ、通常入力は物語入力として混線しないこと

### NFR-005: session分離

- session A の current / cast / archive が session B に混入しないこと
- legacy fallback は read-only 扱いであり、main path を巻き戻さないこと

### NFR-006: prompt肥大化防止

- v1 で使わない重い RPG 要素や旧 inner-galge 語彙を prompt に抱え続けないこと
- 機能分割済み prompt の責務が明確であること

### NFR-007: hotset肥大化防止

- hotset は再開キャッシュであり、攻略メモや全文ログにならないこと
- `今動いている圧 / 未確定情報 / 見えている入口 / 放置で変わること` に絞ること

### NFR-008: GitHub公開時の秘匿性

- 実セッション、個人プレイログ、生ログ、ローカル補助ファイルを Git 管理対象に含めないこと
- 公開物は engine / starter kit として成立すること

### NFR-009: API化時のコスト制御

- prompt の読み込み量、再開時の参照量、hotset の肥大化を抑え、将来の API 利用時に token コストを制御しやすい構造であること

### NFR-010: テスト容易性

- `--prompt-only`、integrity check、PI Player smoke test により、非対話環境でも最低限の回帰確認ができること

### NFR-011: ログ資産化

- 生ログ、章要約、軽量観測メモを分けて扱えること
- current と archive の役割を混同しないこと
- 公開、編集、再利用は将来拡張として整理可能な構造にしておくこと

### NFR-012: 成人向け表現の分離 / 制御

- 成人向け表現を主目的化しないこと
- 必要な関係変化がある場合でも、恋愛、生活、事件の文脈に従って制御すること
- 公開用スターターキットとして過剰な成人寄り語彙を標準要件にしないこと

### NFR-013: Style Layer Preservation

- LIRIA は、場面ごとの文体、温度感、描写密度、会話の間、余韻、視点を制御する Style Layer を持つ
- Style Layer は adult 専用ではなく、日常、生活、恋愛、事件、危機、コメディ、シリアス、親密シーンを含む全体の描写品質を支える
- `style/reference.md`、`style/rules.md`、`style/defaults/*` は保護対象とする
- `prompt/runtime.md`、`prompt/romance.md`、`prompt/combat.md`、`prompt/villain_engine.md` は必要に応じて Style Layer を参照してよい
- 参照は本文模倣ではなく、構造、温度、リズム、余白、視点、描写密度の参照として扱う
- 参照は LIRIA のキャラ、舞台、関係性に変換して使う
- 旧 inner-galge 由来の文言整理時でも、Style Layer を不用意に削除しない

## 4. Terminology and Replacement Policy

LIRIA v1 では、旧 inner-galge 由来の語彙を以下のように置き換える。

- `攻略` → `関係変化`
- `陥落 / 堕とす` → `信頼 / 合意 / 深化`
- `ハーレム管理` → `関係性 / 共同体 / 生活上の利害調整`
- `戦闘` → `危機処理 / 外圧処理`
- `敵` → `事件、外圧、生活や関係を揺らす存在`
- `装備` → `道具 / 仕事道具 / 証拠 / 危機対応ギア`
- `お金` → `生活ランク / 事業状態 / 支払い圧`

標準要件から外すもの:

- 関係変化の主目的を征服や収集に置く発想
- 世界ジャンルの毎回選択
- 詳細な通貨運用
- 拠点投資と施設収入を中心にした経営ゲーム化
- 固定スロット型の装備管理
- 道具の受け渡しを固定的な恋愛トリガーにする設計

### 4.1 Legacy Cleanup Boundary

旧要件整理で削る対象は、旧目的語や旧システム前提である。

- `女を堕とす`
- `ハーレム攻略主目的`
- `攻略報酬`
- `AFFINITY到達ボーナス`
- `魔王討伐標準`
- `装備スロット`
- `G通貨 / 拠点投資 / 施設収入`

一方で、削ってはいけない対象は、描写品質を支える Style Layer である。

- `style/reference.md` の作品 / 作者 / ジャンル別参照
- `style/rules.md` の文体 / 五感 / 余白 / 会話ルール
- `style/defaults/*` の場面別テンプレート
- prompt から Style Layer を選んで場面へ適用する仕組み
- 直接入力を小説的 / 漫画的 / ドラマ的描写へ変換するルール

Style Layer は、旧ハーレム攻略ロジックとは別物であり、LIRIA v1 の文脈へ接続し直して保護する。
- 危機より戦闘ループを主役にする設計
