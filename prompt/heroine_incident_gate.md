# Heroine-Driven Incident Gate

事件 / 依頼 / 違和感を生成する前に通す最小ゲート。
固定一本道の脚本ではなく、ヒロインの日常が揺れる可変の入口を作る。

## Required Before Case Generation

全て短く答えられるまで、incident / case を開始しない。

- `heroine daily life`: 彼女が守ろうとしている生活、場所、人、習慣。
- `daily life at risk`: その日常の何が止まる、失われる、晒される、壊れるのか。
- `why normal windows fail`: 家族、職場、学校、警察、窓口、通常相談だけでは間に合わない / 安全でない理由。
- `why protagonist can enter`: 関係、居合わせ、依頼、能力反応、過去の選択など、主人公が自然に関われる見える理由。
- `shared action`: 主人公とヒロインが同じ場で行う最初の共同アクション。
- `kyun point`: 不安、信頼、照れ、安堵、頼り方が少し変わる恋愛的な揺れ。

## Guardrails

- 主人公ひとりで解ける事件は romance RPG として弱い。ヒロインの感情、行動、判断、失いたくないものが必要になる形へ作り直す。
- `handles` は GM の case card data であり、NPC が会話で列挙する話題ではない。
- 初回相談は、物や記録より先に「誰が心配しているか」「何を失いたくないか」を出す。
- `手順` / `範囲` / `同意` / `守秘` のような運用語を NPC の生台詞にしない。生活語、ためらい、具体行動へ翻訳する。
- 物、記録、ログ、鍵は目的ではない。ヒロインの感情と関係変化を動かす道具として使う。
- 例文は構造理解用であり、具体物、職業、導入、台詞をコピーしない。
- 封筒、鍵、写真、書類、ログ、管理会社などの汎用入口を安易に採用しない。
- Q&A回答、主人公の生活、ヒロインの日常、能力、恋愛の好みから、その卓固有の具体物や出来事を再生成する。
- 同じ導入型、同じ小物、同じ社会窓口が続くなら別案を作る。
- 隠された真相、黒幕、能力の本質は player-facing prose に出さない。見せるのは心配、入口、次に触れる足場だけでよい。
- clue order や解決順を固定しない。プレイヤー選択とヒロインの反応で進み方を変える。
