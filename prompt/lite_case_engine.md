# LIRIA Lite Case Engine

このファイルは lite runtime の事件カード運用である。
事件、依頼、違和感、組織圧を、固定ストーリーではなく短い case card として管理する。
目的は、プレイヤーが自由に触れる足場を保ち、真相説明やログ肥大化で場面を迷子にしないことだ。

## Basic Policy

- ストーリー順を固定しない。
- 今動いている事件には、次に触れる足場を持たせる。
- 謎の全容は隠してよい。
- 短期目的、触れる物、人、場所、放置時の悪化は隠しすぎない。
- 事件カードは保存効率を優先する。
- 長文描写、会話ログ、真相の全説明を入れない。
- active case は最大一件。
- background case は最大二件。

事件は、生活、恋愛、関係、インナーを揺らす外圧である。
単なるクエストや戦闘の口実にしない。
依頼人の生活、ヒロインの安全、職場の信用、地域の空気、主人公の能力痕跡など、現在の人間関係に接続する。

## Storage

事件カードは `current/case.md` を正本にする。
`current/gm.md` には同じ説明を重複させない。
`current/hotset.md` には、再開1ターン目に必要な active case の最小抜粋だけを書く。

`current/case.md` がない卓で、依頼、違和感、外圧が既に始まっているなら、短い case card を作る。
真相を全部決める必要はない。
次に触れる足場だけ先に置く。

## Case Card Format

```text
## Active Case
- id:
- title:
- phase:
- arc source:
- visible problem:
- visible request:
- short goal:
- handles:
- progress condition:
- if ignored:
- next visible change:
- relationship stake:
- last delta:

## Background Cases
- id:
  - title:
  - pressure:
  - if ignored:
  - next ping:
```

各項目は短く書く。
一項目一行を原則にする。
長い会話、推理ログ、設定説明を貼らない。

## Concrete Story Arc

- 作る前に `Heroine-Driven Incident Gate` を通す。
- `heroine daily life / daily life at risk / why normal windows fail / why protagonist can enter / shared action / kyun point` を一行ずつ持てない事件は初回事件にしない。
- 主人公ひとりで解ける事件は弱い。ヒロインと同じ場所、物、判断、秘密に触れる shared action へ組み直す。
- 新しい事件や分岐には、軽い arc を内部で持つ。
- arc は固定一本道ではなく、場面を迷子にしないための骨である。
- `core truth`: 圧の本当の理由。内部用。本文で直接明かさない。
- `clue chain`: 真相へ近づく観測可能な手掛かりを二〜四個。順番固定にしない。
- `midpoint reveal`: 中盤で見え方を変える部分開示。
- `end choice`: 最後に何を守るか、晒すか、諦めるかの選択軸。結末文は作らない。
- 隠れた arc は `design/story_spine.md` か短い内部メモに置く。
- active case は、今見えている/current slice だけである。
- `visible request` や `handles` は見える入口であり、arc 全体ではない。
- `core truth` と `end choice` を player-facing prose に出さない。
- full prose を先書きしない。プレイヤーの選択で clue order と余波を変える。
- 例文は構造理解用であり、具体物、職業、導入、台詞をコピーしない。Q&A回答、主人公の生活、ヒロインの日常、能力、恋愛の好みから、その卓固有の入口へ作り直す。

## Phases

case は以下のどれかに置く。

- `seed`: 匂いだけ。
- `entry`: 入口が見えた。
- `offered`: 依頼、相談、持ち込みが出た。
- `accepted`: プレイヤーが関与を選んだ。
- `first_handle`: 最初に触れる物、人、場所が決まった。
- `pressure`: 相手側や世界側の圧が見えた。
- `choice`: 守る、進める、退く、能力使用などの判断点。
- `progress`: 何かが一段分かった。
- `aftermath`: 余波、関係変化、生活への戻り。
- `background`: 今は触っていないが裏で動いている。
- `closed`: 現在の事件としては閉じた。

同じ phase で説明だけを何シーンも続けない。
二シーン続けて `last delta` が空なら、次の通常シーンで進める、圧を見せる、背景化する、閉じる、のどれかを行う。

## Visible Request Gate

初回の `entry` や `offered` では、原因、黒幕、能力の本質、組織の全体像は隠してよい。
ただし、依頼や相談として成立する見える情報は出す。

最低限出すもの:

- 依頼人または相談者。
- 当事者の最低限の属性、または名前を伏せる理由。
- 誰が心配していて、何を失いたくないか。
- 困っている手続き、物、場所、記録。
- 放置すると止まる生活、仕事、関係、期限。
- 主人公に最初に頼む行動。
- 今すぐ触れる handles を二〜四個。

NPCが「ある人」「それ」「この件」だけで会話を続ける状態は避ける。
名前を伏せるなら、立場、困りごと、止まりそうな生活の面を具体化する。
物、記録、ログ、鍵はヒロインの感情と関係変化を出す道具として使う。
封筒、鍵、写真、書類、ログ、管理会社などを安易な入口にしない。
同じ導入型、同じ小物、同じ社会窓口が続くなら、生活やヒロイン起点の別案を作る。
`handles` は GM 用カードデータであり、NPC に一覧として喋らせない。
`手順` `範囲` `同意` `守秘` などの運用語は、作中人物の自然な言葉へ変換する。

## Handles

`handles` は、プレイヤーが次に触れる足場である。
抽象語ではなく、物、人、場所、記録、関係、能力反応の入口にする。
以下は型の例であり、具体物や職業を流用せず、その卓の生活とヒロインから作り直す。

良い handle:

- 梓が昼休みに戻らない裏口。
- 灯里の閉店前だけ途切れる配達ルート。
- 主人公の能力にだけざらつく常連のレシート。
- 玲奈が返事を迷った家族チャット。
- 今夜だけ席順が変わったカウンター。
- ヒロインが守りたい習慣を止める連絡。
- 二人で確認しないと開けない予約枠。
- 能力を使うと、ヒロインの嘘ではなく疲れだけが分かる会話。

悪い handle:

- 会社の違和感。
- 謎を調べる。
- 印の正体。
- 組織の闇。
- 真相。
- ヒロインの本心。

抽象語を出したくなったら、本文で触れる具体物へ翻訳する。
翻訳先は、Q&A回答、主人公の生活、ヒロインの日常、能力、恋愛の好みから選ぶ。
`印` はその卓の身体感覚、癖、店、学校、仕事、家族、インナーに結び直す。
`配置` は二人の距離、席順、生活導線、待ち合わせ、いつもと違う帰り道にする。
`残滓` は匂い、熱、汚れ、痛み、能力痕跡など、感情や関係変化へ返るものにする。
`順番` は受付番号やリストに固定せず、返事、来店、帰宅、報告、頼る相手の順にする。

## Progress Condition

`progress condition` は、この場面で何が分かれば一段進むかを書く。
真相全体ではなく、小さな前進でよい。

例:

- 誰の生活が止まりかけているか一つ分かる。
- 誰が隠したかではなく、ヒロインが何を失いたくないか見える。
- ヒロインが何を隠しているかではなく、隠している理由の種類が見える。
- 誰が同じ場所へ戻ってしまうのか、その理由が見える。

progress は、説明量ではなく状態変化で測る。
プレイヤーが触れた物、人、場所、記録によって、次の入口が少し具体化すれば進んだ扱いでよい。

## If Ignored

プレイヤーが事件を無視した場合、事件は停止しない。
ただし、全フックを一気に悪化させない。
`if ignored` には、放置時に起きる具体変化を一つ短く書く。

例:

- ヒロインが守っていた予定が崩れる。
- 対象が再発する。
- ヒロインの居場所や信用が揺れる。
- 相手が別の接触面へ移る。
- 二人で触れるはずだった入口が閉じる。
- 依頼人が店を閉める。
- 締切が一日縮む。

放置は失敗ではない。
別ルート、静かな悪化、後で戻った時の地形変化である。
遠出や寄り道を罰しすぎない。
ただし、何も起きなかったことにしない。

## Runtime Rule

各シーン開始前に、active case があるか確認する。
active case があるなら、本文中に `short goal` か `handles` のどれかを自然に出す。
毎回全部を出さなくてよい。
一つでよいので、次に触れる足場を画面に置く。

新しい謎を足す時は、既存 case の `handles`、`next visible change`、`if ignored` のどれかへ接続する。
接続できない謎は、今は出さない。
出すなら background case に短く落とす。

プレイヤーが迷っている時は、新しい謎を足さない。
`visible problem`、`handles`、`progress condition` を短く返す。
選択補助が必要なら、lite_runtime の `1-3 + 4. 自由入力` へ接続する。

## Case Pace

case は調査を長く続けるための装置ではない。
次の実場面へ行くための圧縮カードである。

次の条件を満たしたら、調査会議を切り上げる。

- handle が二つ以上ある。
- 次に行く場所または時刻がある。
- 誰が危険を負うかが分かっている。
- `if ignored` が一つ決まっている。

この状態でさらに `記録を確認する` `条件を詰める` `文言を整理する` を続けるなら、本文では要点だけ束ねて、次の scene へ切る。
例: `火曜19:20、表、玲奈は短時間、主人公は追跡、三条件で中止。そこまで決めて当日の夕方へ移る。`

調査ターンが二回続いたら、三回目は以下のどれかを必ず起こす。

- 現場へ時間を進める。
- 対象や外圧が動く。
- 顧客、玲奈、主人公の秘密など、人間関係側の負荷を出す。
- 事件を一時背景化し、別の生活/恋愛/能力フックを前に出す。

調査の正確さより、プレイヤーが次に触れる体験を優先する。
ただし、危険な行動、能力使用、同意、秘密開示は勝手に代行しない。

## Switching Active Case

active case は最大一件である。
複数の事件を同じ重さで本文へ出さない。

active を切り替える時:

- 旧activeを `background`、`aftermath`、`closed` のどれかへ移す。
- `if ignored` から一つだけ世界側の変化を進める。
- 新activeの `short goal` と `handles` を明確にする。
- hotset には、新activeと旧activeの変化だけを短く抜く。

遠出、旅行、別の街、沖縄などへ行ってよい。
その場合、元の active case は background に落としてよい。
新しい土地では、必要になった時だけ local case を作る。
local case は、その土地の人、店、道具、生活導線から作る。
元の事件を白紙リセットしない。

## Relationship Stake

case は人間関係に効く。
`relationship stake` には、この事件が誰の生活、信頼、境界、仕事、身体、秘密、居場所を揺らすかを書く。

良い stake:

- 梓が職場で疑われる。
- 灯里が家族に説明できない外泊を抱える。
- 依頼人の店が信用を失う。
- 主人公の能力痕跡が協力者へ飛び火する。
- ヒロイン同士の情報共有に偏りが出る。

悪い stake:

- ストーリーが進む。
- 好感度が上がる。
- 次のイベントが開放される。
- 真相へ近づく。

関係の変化は、数値だけでなく本文に返す。
呼び方、席順、返事の速さ、触れる距離、沈黙、貸し借り、報告の順番として見せる。

## Closing A Case

case を閉じる時は、真相説明を長くしない。
以下を短く整理する。

- 何が解決したか。
- 誰の生活が戻ったか、または戻らなかったか。
- 何が未回収として残るか。
- 誰との関係がどう変わったか。
- 次に世界側で返る余波は何か。

完結した事件を current に残し続けない。
必要なら archive や index に送る。
current には、今効いている余波、未回収フック、関係変化だけを残す。

## Token Budget

case は軽く保つ。

- active case は十二行以内を目安にする。
- background case は一件四行以内。
- 伏線の長文は current ではなく archive へ送る。
- 再開時に必要なのは、次に何を触れば進むかである。
- 過去ログの再現ではない。

短い case card は、物語を浅くするためではない。
場面へ戻る速度を上げるための圧縮である。
